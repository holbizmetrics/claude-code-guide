# ChatMerge — Lightweight TRIAD + KG Audit (2026-05-11)

**Scope:** Audit ChatMerge before applying my 7 proposed improvements
from the mass-scale use session of 2026-05-11 (110 inputs → 68 streams
during chat-archive build).

**Method:** TRIAD pass (Gate / Engine / Mirror) + KG fault-class scan
on the 7 proposed candidates. Goal: catch substrate-priors slip and
distinguish "real bug" from "consequence of how I used the tool."

**Outcome:** **2 of 7 candidates partially invalidated; 1 surfaced as
SRL-slip on my part; 4 stand as legitimate fixes.** Falsifier
pre-commits documented per surviving fix.

---

## TRIAD pass

### Gate (does the tool gate inputs correctly?)

✓ Accepts directory or file globs or explicit files
✗ **No validation of input file types** — ChatMerge silently processes
  build artifacts, README files, etc. as if they were chat exports.
  Real-world impact: my staging passed `*.csproj.FileListAbsolute.txt`
  files (MSBuild artifacts) into ChatMerge as if they were chats.
  Mild risk; outputs would be garbage but no crash.

### Engine (does the merge actually do what's claimed?)

✓ Detects progressive exports and groups them into streams correctly
  (verified empirically: produced 68 streams from 110 inputs)
✓ Cross-deduplicates message content within stream
? **Stream-grouping heuristic is opaque.** Source uses
  `GetCommonPrefix(filenames)` to derive stream identifier — works for
  semantically-named files (`chat-export-x-1.txt`, `chat-export-x-2.txt`
  → stream "chat-export-x"). **Breaks when inputs lack shared prefix**
  (see Mirror finding below).

### Mirror (does the output faithfully reflect input semantics?)

✗ **CRITICAL FINDING: the catchall `"merged"` stream name is opaque.**
  When inputs in a group don't share a common filename prefix,
  `GetCommonPrefix` produces empty string, defaulting to `"merged"`.
  This produces output files literally named `merged_merged.txt`.

  Investigation: read first 50 lines of `merged_merged.txt` (9764 lines,
  581K chars) — it's a single conversation lineage (a PCL boot
  transcript). NOT a misclassification of multiple conversations into
  one. So Mirror is functionally correct (one stream = one
  conversation) but semantically opaque (the user can't tell what
  conversation this stream contains without opening the file).

  **Resolution:** Mirror is correct in content; opaque in label.
  Improvement candidate: when common prefix is empty, fall back to
  the largest input's filename or first non-empty content line.

## KG fault-class scan over 7 proposed candidates

| # | My proposed fix | KG fault check | Verdict |
|---|---|---|---|
| 1 | Catchall "merged" stream too aggressive | **SRL-slip:** my hash-prefix staging caused it. ChatMerge stream-grouping uses `GetCommonPrefix(filenames)`. My inputs `<hash>__<name>.txt` share NO prefix between distinct hashes, so common-prefix calculation degraded. Real bug: opaque label, not aggregation behavior. | **PARTIALLY INVALIDATED** — fix needs rescoping |
| 2 | No `--dry-run` flag | Source confirms only `--single, -s` and `-o` exist. Activation-vs-Existence: not undocumented. | **CONFIRMED LEGIT** |
| 3 | No summary stats at end | Source confirms only per-stream output. | **CONFIRMED LEGIT** |
| 4 | Confusing input-to-output handling | **SRL-slip:** ChatMerge does NOT consume inputs (lines 156-164 write new files; no `File.Delete` or `File.Move` in source). My earlier "30 inputs disappeared" count was likely wrong (counted differently in two passes). Re-verify before claiming. | **NOT CONFIRMED — investigate first** |
| 5 | Hash-prefix carry-through to output names | **SRL-slip cousin:** hash-prefix was MY addition for staging dedup. ChatMerge faithfully passes through whatever filename prefix exists in inputs. Suggest a `--strip-prefix <regex>` is a real ergonomic improvement, but the carry-through itself isn't a bug. | **CONFIRMED LEGIT** as ergonomic, not bug |
| 6 | No content-hash dedup built in | Activation-vs-Existence check: source has no SHA-256/MD5 import, no hash-based dedup. Not undocumented. | **CONFIRMED LEGIT** |
| 7 | No `--min-overlap-lines N` flag | Source uses LCS-based overlap detection without configurable threshold. Activation-vs-Existence: not exposed. | **CONFIRMED LEGIT** |

**Net:** 4 confirmed-legit + 1 ergonomic + 1 needs investigation + 1 SRL-slip.

## Falsifier pre-commits per surviving fix

For each fix that proceeds, the falsifier states what we expect to see if
the fix is real (and therefore what we'd see if it was vacuous).

### Fix #1 (rescoped) — Better stream label when common prefix is empty

**What changes:** When `GetCommonPrefix` returns empty/whitespace, fall
back to the largest input's filename (without extension) as the stream
label. Output becomes `<largest-input-name>_merged.txt` instead of
`merged_merged.txt`.

**Falsifier:** With my chat-archive corpus (which had hash-prefixed
inputs causing many empty common-prefix groups), re-running ChatMerge
should produce **at least 5 stream files with semantically-meaningful
names** instead of the single catchall `merged_merged.txt`. If only
0-2 such names emerge, the fix isn't load-bearing in practice.

### Fix #2 — `--dry-run` flag

**Falsifier:** On the chat-archive corpus, dry-run output should match
the actual run's stream count (68) and predicted output sizes within
±5%. If predictions diverge significantly, the dry-run isn't faithfully
modeling the real merge logic.

### Fix #3 — Summary stats

**Falsifier:** Summary should report ≥4 distinct numbers I'd actually
want: total inputs, total streams, bytes saved, %dedup. If any of these
4 turns out to be uninterpretable for users, scope the fix tighter.

### Fix #4 — INVESTIGATE FIRST

**Action:** Before changing anything, re-run ChatMerge on a clean staged
input set; verify whether inputs are or aren't consumed/renamed. Then
decide if a `--keep-inputs` flag (or documentation) is needed.

### Fix #5 — `--strip-prefix <regex>`

**Falsifier:** With regex `^[a-f0-9]{16}__` matching my hash prefix,
output filenames should drop the prefix while semantic suffix is
preserved. If regex matching causes name collisions or strips MORE than
intended, fix is over-broad.

### Fix #6 — Content-hash dedup preflight (`--content-hash-dedup`)

**Falsifier:** On a synthetic input with 10 byte-identical files
(zero-padded with whitespace differences? real bug case),
content-hash should reduce to 1 input before stream grouping. If it
still produces 10 separate inputs, hash isn't doing what's claimed.

### Fix #7 — `--min-overlap-lines N`

**Falsifier:** With N=100 (high), the catchall "merged" stream should
break into MORE individual streams (overlap threshold harder to meet).
With N=5 (low), MORE streams should collapse. If neither direction
changes the count, the threshold isn't load-bearing in this corpus.

## Methodology notes

This audit took ~30 minutes to write, gated 7 proposed fixes by the
methodology, surfaced **2 SRL slips** I would have shipped as bugs
otherwise (#1, #4), and produced falsifiers for the surviving 5. The
fix work proceeds with clear gates instead of working off intuition.

This is what the methodology is for at infrastructure scale — not just
findings but tooling decisions get audited too. Consistent with the
bootstrapping pattern: applying KG to the tool that built the chat
archive that future sessions will consume.

**Activation-vs-Existence (Q#19 from research-questions-queue) was
explicitly tested and did NOT fire — none of my "missing features"
were undocumented existing flags.** That's a small-scale validation
that the fault class is identifiable in practice.
